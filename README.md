# Database DSL Compiler

This repository contains a course project focused on building a compiler for a small domain-specific language (DSL) used to describe the structure of a database. The compiler is implemented in Java with ANTLR and currently translates the custom language into PostgreSQL-oriented output.

The web application and the API exist to make the compiler usable from a browser, but the compiler is the center of the project and the main academic deliverable.

The original project brief is in `docs/instructions.md`. It asks for:

- a simple language for defining a database and its tables
- a web interface where users can write and compile that language
- SQL generation from the custom language
- a second text file describing the database structure
- a final written report

## Main Goal

The main goal of this repository is to design and implement a compiler that:

- accepts a simple database-description language
- parses it with ANTLR
- validates the input
- transforms it into PostgreSQL-oriented output

The repository also provides the infrastructure around that compiler so it can be tested from a browser:

- a Spring Boot API that exposes compilation over HTTP
- an Astro + React frontend that works as a small IDE/editor

## Current End-to-End Flow

`Web editor -> Spring Boot API -> Java/ANTLR compiler -> generated PostgreSQL SQL`

## Current Scope

This repository includes:

- ANTLR grammar for the current version of the DSL
- Java compiler facade for invoking the parser
- SQL generation for database creation, table creation, fields, and simple relationships
- HTTP API for sending source text and receiving SQL or errors
- web interface for writing source code and compiling it

## Architecture

The repository is organized around the compiler. The API and web app are wrappers around it.

### 1. Compiler

Location: `packages/compiler`

This is the most important package in the repository.

- Built with Java 17 and Maven
- Uses ANTLR 3.5.2
- Main entrypoint: `dev.andyalv.compiler.CompilerFacade`
- Grammar source of truth: `packages/compiler/src/main/antlr3/dev/andyalv/compiler/root.g`

What it does:

- tokenizes and parses the DSL
- applies syntax and some semantic validation
- builds the SQL output for the recognized structures

Important rule:

- Do not edit `packages/compiler/target/generated-sources/antlr3/*`
- Those files are generated by Maven from `root.g`

### 2. API

Location: `apps/api`

- Built with Spring Boot
- Main endpoint: `POST /compile`
- Validates the request body
- Calls `CompilerFacade` from the compiler package
- Returns either generated SQL or an error message as JSON

Important detail:

- the API depends on the local Maven artifact `dev.andyalv:compiler:1.0-SNAPSHOT`
- because of that, the compiler must be installed locally before the API can run on a fresh machine

### 3. Web application

Location: `apps/web`

- Built with Astro, React, TypeScript, Tailwind CSS, and CodeMirror
- Main page: `apps/web/src/pages/index.astro`
- Main UI component: `apps/web/src/components/compiler-workbench.tsx`

What it does:

- provides an editor for the custom language
- sends the source code to the API
- displays either generated SQL or compilation errors

### 4. Dev services

Location: `packages/dev-services`

- Contains a local PostgreSQL 17 Docker Compose setup
- Useful for manual experiments

## The Language

The project brief in `docs/instructions.md` describes the language idea in Spanish, but the current implementation uses English keywords.

Current keywords include:

- `create database`
- `with`
- `table`
- `start`
- `end`
- `relationship`
- `closecon`
- data types: `integer`, `numeric`, `string`, `date`

Example source accepted by the current implementation:

```text
create database enterprise
with enterprise
table depto
    start
       name string 40
       department_code string 10
       number integer
       responsabilities string
    end
table empleado
    start
       name string 40
       age integer
       birth_date date
       salary integer
       relationship depto
    end
closecon
```

Example output:

```sql
CREATE DATABASE enterprise;
\c enterprise
CREATE TABLE IF NOT EXISTS depto (depto_key SERIAL PRIMARY KEY, name VARCHAR(40), department_code VARCHAR(10), number INTEGER, responsabilities TEXT);
CREATE TABLE IF NOT EXISTS empleado (empleado_key SERIAL PRIMARY KEY, name VARCHAR(40), age INTEGER, birth_date DATE, salary INTEGER, fk_depto INTEGER REFERENCES depto(depto_key));
\q
```

Current behavior to be aware of:

- `string` without precision becomes `TEXT`
- `string 40` becomes `VARCHAR(40)`
- `relationship depto` only works if `depto` was defined earlier in the same source
- the output includes `\c` and `\q`, which are `psql` commands rather than plain SQL statements

```sh

```

## Running the Project

Turbo is the root-level runner for the workspace applications.

### Recommended order

1. Install JavaScript dependencies.
2. Install the compiler artifact into the local Maven repository.
3. Run the applications with Turbo from the root.

### 1. Install the compiler locally

This step is required before running the API on a fresh machine, and again after changing the grammar.

```sh
cd packages/compiler
mvn install
```

### 2. Run both apps from the root with Turbo

```sh
bun dev
```

This runs the root Turborepo pipeline and starts the workspace `dev` scripts concurrently.

### Full recommended setup sequence

```sh
bun install
cd packages/compiler
mvn install
cd ../..
bun dev
```

## Running Individual Parts

### Web only

```sh
bun --filter web dev
```

### API only

```sh
bun --filter api dev
```

### Compiler only

```sh

```

The compiler package is a Maven package, so it is not started through a Bun workspace script like the web and API apps.
